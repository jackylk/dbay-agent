计算隔离层
Layer 1 · Kata VM（最强）

容器逃逸防护

  • 每个 Pod = 独立 Kata 轻量 VM，独立内核
  • 宿主机资源对容器完全隔离（CCI 官方确认）
  • 用户代码无法突破 VM 边界
  • 替代传统 seccomp/AppArmor 的硬边界
Layer 2 · IAM + namespace

租户权限隔离

  • 每租户独立 CCI namespace，IAM 细粒度授权
  • tenant-a 的 AK/SK 只能操作 tenant-a-ns
  • OBS STS 临时凭证 TTL = Job 生命周期
  • Celeborn mTLS + appId ACL 双层隔离
Layer 3 · mTLS per-job

身份认证（替代 NetworkPolicy）

  • Cert Manager 为每个 Job 签发短期证书
  • Driver ↔ Executor ↔ Celeborn 全链路 mTLS
  • 没有证书的连接在应用层被拒绝
  • NetworkPolicy 不支持 Kata，此层为精细化补充
接入 & 平台组件安全层
Layer 4 · 接入安全

Gateway 防线

  • IAM Token 强制验证，无 Token 直接拒绝
  • Spark conf 白名单：禁止 spark.kubernetes.* 系列参数
  • 禁止 shell 注入，参数严格类型检查
  • 限流：per-user QPS 限制，防资源滥用
Layer 5 · 镜像安全

Image Pipeline

  • 禁止用户直接推镜像，只能提交 requirements.txt
  • Kaniko 在隔离 Pod 中构建，非 privileged
  • Trivy 扫描：Critical 级漏洞阻断构建
  • Cosign 签名，Controller 只允许运行签名镜像
  • CCI 不支持第三方 registry，强制 SWR
Layer 6 · 平台组件零信任

东西向防护（原方案缺失）

  • Job Gateway：只能写 CRD，不能调 CCI API
  • Spark Operator：只能操作 SparkApplication + VK Pod
  • Ray Controller：只能操作 RayJob + VK Pod
  • Virtual Kubelet：只能调 CCI API，不能操作 CCE 真实节点
  • 组件间全部 mTLS，禁止明文内部通信

组件职责
每个组件的核心职责、改造程度和 RBAC 权限边界

Job Gateway

自研
  • IAM Token 验证，无 Token 拒绝
  • Spark conf 白名单过滤（禁 spark.kubernetes.*）
  • 参数类型检查，防 shell 注入
  • 创建 SparkApplication / RayJob CRD
  • per-user 限流，防资源滥用
RBAC: create SparkApplication/RayJob · 禁止调 CCI API / RDS

Spark Operator

社区版不改
  • Watch SparkApplication CRD，管理状态机
  • 调 CCE K8S API 创建 Driver Pod（VK 透明调度到 CCI）
  • 处理重试、超时、Job 生命周期
  • 社区版完整保留，不 fork
RBAC: watch/update SparkApplication · create Pod on VK node only

Ray Controller

自研
  • Watch RayJob CRD
  • 创建 Head Pod（VK → CCI），注入 Redis GCS 配置
  • 创建 Worker Pod 初始数量
  • 管理 Ray 集群生命周期，Job 结束清理
RBAC: watch/update RayJob · create Pod on VK node only

Virtual Kubelet HA

华为云插件
  • 以虚拟节点形式接入 CCE，透明转发 Pod 到 CCI
  • 双副本 + leader election，消除单点
  • 资源配额随弹性 Pod 数线性配置（需提前规划）
  • 需购买 VPCEP（CCI 2.0 要求）
  • 仅支持 VPC 网络模式 CCE 集群
RBAC: 调 CCI API · 禁止操作 CCE 真实节点

Spec Normalizer

自研 Webhook
  • MutatingAdmissionWebhook，拦截所有 Pod 创建
  • 将 Pod 规格预对齐到 CCI 固定规格（2C4G / 4C8G / 8C16G…）
  • 避免 CCI 自动规整导致的资源浪费（如 3C6G → 4C8G）
  • 强制 requests = limits（CCI 要求）
作用于所有调度到 VK 节点的 Pod

Signal Bridge Sidecar

自研 · 注入 Driver
  • 随 Driver Pod 以 Sidecar 方式部署
  • 监听 Spark Dynamic Allocation 扩缩信号
  • 将信号转发给 Spark Operator（HTTP）
  • 替代 Driver 直接调 K8S API 的原始机制
  • Driver 零 K8S 权限，安全边界清晰

Cert Manager

自研 / 社区
  • 每个 Job 启动时签发短期 mTLS 证书
  • TTL 与 Job 生命周期绑定，自动吊销
  • Driver、Executor、Celeborn 各持独立证书
  • 替代 NetworkPolicy 无法使用的精细网络控制

Celeborn（RSS）

社区 + 改造
  • 部署在 CCI（Kata VM），与 Executor 同安全域
  • 每个 Worker 挂载独立 EVS SSD PVC（替代 HostPath）
  • emptyDir 内存层作热数据缓冲
  • mTLS 双向认证 + appId ACL 多租隔离
  • per-app quota 防磁盘 DoS

metrics-exporter Sidecar

替代 DaemonSet
  • 自动注入到 Driver / Head Pod
  • 暴露 JVM metrics + 资源用量
  • Prometheus 通过 Pod IP 直接 scrape
  • DaemonSet 在 CCI 不可用，此方案完全替代
  • 粒度更精细：per-Job 级别指标
CCI 能力核查
基于华为云官方文档的全面核查，每项均有对应的方案处理
能力项 CCI 支持情况 严重度 V4 方案处理
NetworkPolicy 不支持（Kata 容器限制，官方文档明确） 严重 三层替代：Kata VM + IAM namespace + mTLS per-job
HostPath 卷 不支持（安全性限制，官方文档明确） 严重 Celeborn 改用 EVS SSD PVC (per-worker) + emptyDir 内存缓冲
DaemonSet 不支持 严重 metrics-exporter sidecar 自动注入 Driver/Head Pod
privileged 容器 不支持（宿主机完全隔离） 中等 用 SecurityContext capabilities 细粒度替代（SYS_ADMIN 按需）
Pod 规格规整 自动向上取整到固定规格（6C15G→8C16G） 中等 Spec Normalizer Webhook 提交前预对齐，成本可控
第三方 registry 不支持，只能用 SWR 中等 Image Pipeline 强制推送到 SWR，Controller 只允许 SWR 镜像
镜像单层大小 解压后不超过 20G 中等 Spark/Ray 基础镜像分层优化，依赖层独立，单层控制在限制内
VK 默认单副本 单实例无 HA，需手动配置多副本 中等 明确配置双副本 + leader election，资源按 Pod 数公式计算
VPCEP 必须购买 CCI 2.0 需要，按时计费 中等 纳入成本预算，提前购买规划
子网冲突约束 不能与 10.247.0.0/16 重叠 中等 网络规划阶段强制检查，集群创建后不可修改
emptyDir（内存型） 支持，从 Pod 内存分配 — Executor shuffle 写缓冲，/dev/shm 扩容，建议设为内存的 50%
EVS 云硬盘 PVC 支持（高 IO 型 / 通用型） — Celeborn Worker 持久化存储，per-worker 独立 PVC 隔离
SFS Turbo 挂载 支持 — 备选：EVS 性能不足时，用 SFS Turbo 作 Celeborn 存储
OBS 挂载 / Spill 支持 — Ray Object Spill，Celeborn 溢出兜底，STS AK/SK 隔离
GPU Pod（V100/T4） 支持 — Ray GPU Worker 可用，CUDA 版本需匹配 460.106 或 418.126
InitContainer / Sidecar 支持 — Signal Bridge、metrics-exporter 均以 sidecar 形式注入
Service / DNS 互通 支持（需开启 Networking 选项） — Driver ↔ Executor 服务发现正常，VK 安装时选择开启
IAM 细粒度权限 支持，namespace 级别 — 多租隔离核心机制，替代部分 NetworkPolicy 的访问控制作用
POC 计划
按风险优先级排序，全部通过后才进入正式开发。每项均有明确通过/失败标准，失败时有降级方案。
P0

EVS PVC on CCI — Celeborn shuffle 吞吐基准测试

HostPath 不可用，Celeborn 必须改用 EVS。EVS 延迟比本地 NVMe 高，必须实测确认是否满足 Spark shuffle 性能要求。这是最大的物理约束，直接决定 Celeborn 的存储方案。

通过标准：EVS 高 IO 型单 Worker 顺序写 ≥ 200MB/s，P99 延迟 ≤ 5ms 失败降级：改用 SFS Turbo（共享高性能文件系统），或多块 EVS 条带化
2 天
P1

Virtual Kubelet + CCI 网络连通性验证

验证 CCE 安装 VK 插件后，Pod 调度到 CCI 的全链路网络。包括 Pod 间通信、Service DNS 解析、跨 namespace mTLS 连接、CCI 是否暴露 metadata endpoint（决定是否需要额外封锁）。

通过标准：Driver Pod 能通过 Service DNS 找到 Executor Pod，mTLS 握手成功，metadata endpoint 不可达（CCI 已封锁） 失败降级：metadata 可达则需加 iptables 规则封锁（确认 CCI 是否支持）
2 天
P2

Spark Operator + VK 端到端

提交真实 SparkApplication CRD，观察完整链路：Driver Pod 创建→Executor 注册→Dynamic Allocation 信号（Signal Bridge）→Executor 扩缩→Job 完成→Executor 回收。验证 Signal Bridge sidecar 的信号传递是否正常。

通过标准：一个 SparkPi 作业完整跑通，DA 触发 Executor 扩容并在 idle 后缩容，Celeborn shuffle 成功读写 失败降级：排查 DA 信号链路，必要时改为静态 Executor 数量（放弃 DA）
3 天
P3

Ray Head + GCS HA 验证

在 CCI 上运行 Ray Head + Worker，GCS 外置到 Redis。强制 kill Head Pod，验证 Head 重启后 Job 状态从 Redis 恢复，Worker 重新注册，未完成的 Task 继续执行。

通过标准：Head 重启耗时 ≤ 30s，Job 状态完整恢复，Worker 重新注册后 Task 继续执行，无数据丢失 失败降级：如 GCS HA 不稳定，采用 Checkpoint 到 OBS 的 Job 级容错
2 天
P4

Spec Normalizer + 规格规整验证

提交边界规格 Pod（3C6G、5C10G、7C14G），验证 Spec Normalizer Webhook 的预对齐是否覆盖所有情况，以及 CCI 实际规整结果是否与预期一致。验证成本计算的可预测性。

通过标准：所有测试规格经 Normalizer 预对齐后，CCI 不再触发额外规整；成本计算误差 ≤ 5%
1 天
P5

镜像预拉取 + 冷启动时间测量

在 CCI 节点预拉取 Spark/Ray 镜像，测量有缓存 vs 无缓存的 Pod 启动时间 P99。给出准确的冷启动基线数据，决定是否需要进一步的启动优化。

通过标准：有镜像缓存时 Pod Running P99 ≤ 15s（含 Kata VM 启动开销） 失败降级：如超出，评估 ImageSnapshot 功能（CCI 支持）进一步加速
1 天
